### 3. DETERMINISTIC VALIDATORS (qa_engine/validators/)

✓ validate_schema_compliance() - Hard fail if required fields missing

✓ validate_format_compliance() - IELTS Part structure validation

**qa_engine/validators/bias_safety.py**

- pass/fail boolean
- confidence score (0-1)
- checks_performed (dict of sub-check results)
- issues_found (human-readable)
- severity (info/warning/error)

Total file size: ~350 lines


### 4. QA ORCHESTRATOR (qa_engine/orchestrator.py)
✓ run_qa_validation_pipeline() - Runs all validators in sequence
✓ Composite quality score (0-100) weighted average across gates
✓ Auto-stop on hard failures (schema, format)
✓ Recommendation logic (accept/review/reject)
✓ Next stage determination (AUTO_VALIDATED/EXPERT_REVIEW_PENDING/REJECTED)

Decision tree:
- All required gates pass + quality_score >= 85 → ACCEPT (auto-activate)
- Quality_score 70-84 + warnings → REVIEW (expert human review)
- Quality_score < 70 or hard fails → REJECT

File size: ~200 lines


### 5. ITEM LIFECYCLE MANAGER (qa_engine/lifecycle.py)
✓ create_lifecycle() - Initialize DRAFT item
✓ transition_item() - State machine with validation
✓ apply_validation_report() - Auto-advance based on validation results
✓ get_event_history() - Full audit trail
✓ get_active_duration_days() - Calculate active lifespan

Valid state transitions (enforced):
  DRAFT → AUTO_VALIDATED → EXPERT_REVIEW_PENDING → EXPERT_APPROVED → 
  SHADOW_TESTING → SHADOW_CALIBRATED → ACTIVE → MONITORING → 
  DRIFT_FLAGGED → RETIRED / REJECTED

Each transition logged with:
- Timestamp
- From/to status
- Reason
- Triggered by (user_id or system)
- Decision data (metrics, scores, flags)

## ============================================================================
## USAGE PATTERNS
## ============================================================================

### Pattern 1: Generate → Validate → Auto-Activate (Most Common)
POST /orchestrate (generate_item)
  ↓
POST /orchestrate (validate_item)
  ↓
If quality_score >= 85 & all gates pass:
  → Item auto-moves to ACTIVE
  → Appears in question bank immediately


### Pattern 2: Generate → Validate → Expert Review (For Complex Items)
POST /orchestrate (generate_item)
  ↓
POST /orchestrate (validate_item)
  ↓
If quality_score 70-84 or warnings exist:
  → Item moves to EXPERT_REVIEW_PENDING
  → Curriculum manager reviews via dashboard
  → Approves or rejects manually


### Pattern 3: Monthly Drift Detection (Automated Job)
Scheduled job (monthly):
  ↓
For each ACTIVE item:
  → Fetch monitoring metrics from past 30 days
  → POST /orchestrate (monitor_drift)
  → If alert_triggered:
    - Flag for curriculum review
    - Auto-retire if breach threshold exceeded


### Pattern 4: Research Export (Quarterly)
Researcher requests data export:
  ↓
Query item lifecycle database
  ↓
Aggregate ResearchExportRecords
  ↓
Anonymize (remove user IDs, names)
  ↓
Export as CSV/JSON/Parquet

### To Test Validators Manually:

```python
from backend.qa_engine.schemas import QuestionItem, Pathway, TargetLevel
from backend.qa_engine.orchestrator import run_qa_validation_pipeline

item = QuestionItem(
    item_id="test_001",
    spec_id="spec_001",
    instruction="Tell me about yourself.",
    pathway=Pathway.CEFR,
    target_level=TargetLevel.CEFR_B1,
    task_type="part_1",
    domain="everyday",
    skill_focus="mixed",
    generated_by="template",
)

report = run_qa_validation_pipeline(item)
print(f"Quality Score: {report.quality_score}")
print(f"Passed: {report.overall_pass}")
for result in report.validation_results:
    print(f"  {result.gate}: {result.passed}")
```
